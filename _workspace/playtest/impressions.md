## 카테고리: 메인 메뉴 / 정보 구조 (IA)

### #001 메인 메뉴 정보 과다
- **언제/어디서**: 로그인(메인) 화면, 첫 플레이 시점
- **감정**: 정보 과다 (메뉴 7개)
- **사용자 요구 (날 것)**:
  - 첫 진입 시 노출은 **게임 시작 / 세이브 코드** 2개면 충분
  - **연구 / 도감 / 기록 / 카오스** → 신규 "**세이브 관리**" 메뉴를 만들어 그 안으로 수납
  - "세이브 관리" 메뉴는 **한 번이라도 플레이한 뒤에야** 노출 (첫 플레이 전엔 숨김)
  - **디버그 전투** → 평소엔 숨김. 메인에서 **특수 커맨드 입력** 시 노출하거나, 아니면 구석으로 치움
  - **전생 / 보스 클리어 수치** → "**기록**" 메뉴 안으로 이동
- **메모**: 사실상 메인 메뉴를 "처음엔 미니멀, 진행할수록 확장"되는 점진적 노출(progressive disclosure) 구조로 재편하자는 방향.

### #002 연표(챕터) 선택 화면 정보 과다 + 감성 문구 제거
- **언제/어디서**: 연표 선택 화면 (게임 시작 직후)
- **감정**: 정보 과다, 군더더기 문구 거슬림
- **사용자 요구 (날 것)**:
  - 상단 문구 `"한 연표를 골라 — 그 시대의 단면으로 깃들어 간다"` **삭제** (불필요)
  - 화면 명칭/성격을 "연표 선택" → **"챕터 선택"** 으로 (내부 코드/변수는 "연표" 단어 유지 OK, 표시만 챕터)
  - 정렬: **해금 순서대로** 개편
  - 노출 정보를 다음 정도로 **최소화**:
    - `1장` (챕터명)
    - `제한 시간`
    - `도감 해금 정도(%)`
  - 진입 버튼 텍스트: → **"선택"** 으로 건조하게
- **메모**: #001과 같은 결 — "표시는 미니멀·건조하게, 내부 용어는 그대로". 감성 카피 제거 + progressive disclosure 흐름의 연장선.

### #003 캐릭터(종족) 선택 화면 — 명칭/문구/표시 정보 정리
- **언제/어디서**: 종족(캐릭터) 선택 화면
- **감정**: 정보 과다 + 죽은 정보(MP) 노출
- **사용자 요구 (날 것)**:
  - 하단 문구 `"이 시대에 깃들~"` → **"플레이할 캐릭터를 선택해주세요."**
  - 화면 명칭 "종족 선택" → **"캐릭터 선택"**
  - 각 종족의 **플레이버 텍스트 수정 필요** (별도 작업 트랙으로 묶일 듯 — 한국어 텍스트 다듬기와 연동 가능)
  - 표시 정보에서 빼야 할 것:
    - **MP**: 이미 게임에서 없어진 정보 → 화면뿐 아니라 **데이터/스키마까지 정리 필요** (기술 부채)
    - **덱 / 시드 카드**: 선택 단계에서 중요하지 않은 정보 → 노출 축소/제거 검토
    - **HP**: 이것도 사실 선택 단계에선 중요하지 않은 정보
- **메모**: 선택 화면에 남길 "정말 중요한 정보"가 무엇인지 종합 단계에서 역으로 정의 필요(현재 거의 다 "안 중요"로 표시됨 → 무엇을 보여줄 것인가가 빈칸). MP 제거는 표시 정리 + 데이터 마이그레이션 두 갈래.
- **연결**: 플레이버 텍스트 수정은 진행 중인 [[project_krtext_overhaul]] 트랙과 합류 가능.

### #004 카오스 선택 창 — 조건부 노출 + 문구 정리
- **언제/어디서**: 런 시작 전 카오스 선택 창
- **감정**: 불필요한 노출 + 군더더기/낡은 표현
- **사용자 요구 (날 것)**:
  - 카오스가 **1개도 해금되지 않았으면 카오스 선택 창 자체를 노출하지 않음** (조건부 스킵)
  - 안내 문구 `"카오스가~ 기록된다"` → **"카오스 레벨을 선택해주세요. 카오스 수치가 높을수록 원래 세계에서 멀어집니다."**
  - "연표" 표현 삭제 방침에 맞춰, 최고 기록 라벨 → **"이 챕터의 최고 기록"** 으로 변경
- **메모**: #001~#003과 동일한 결 — 점진적 노출(해금 전 숨김) + "연표" 용어 표시상 제거 + 건조한 카피. 카오스 톤 규칙([[project_chaos_system_spec]]: 플레이어 노출 톤="원래 세계에서 멀어진다", "위기 협약" 등 내부 용어 UI 금지)과 일관됨.

### #005 런 시작 직전 브리핑(미션) 팝업 신규
- **언제/어디서**: "런 시작" 버튼을 누른 직후
- **감정**: (개선 제안) 진입 직전에 목표를 명확히 환기시켜 달라
- **사용자 요구 (날 것)**:
  - "런 시작"을 누르면 **간단한 확인 팝업**을 하나 띄움
  - 이 팝업이 실질적으로 **미션을 알려주는 자리**로 기능하는 게 낫다
  - 팝업 내용:
    - `1장` (챕터)
    - `제한 시간`
    - `미션` (목표를 다시 명시)
    - 카오스를 골랐다면 → **선택한 카오스 정리해서 함께 표시**
- **메모**: 앞선 #002~#004에서 진입 화면들을 "미니멀"하게 깎은 만큼, 깎여나간 핵심 정보(목표/제한시간/카오스)를 **이 마지막 브리핑 팝업에서 한 번에 재확인**시키는 구조. 즉 "각 단계는 가볍게, 시작 직전에 종합 브리핑"이라는 온보딩 플로우의 종착점. → #003에서 남긴 "선택 화면에 뭘 보여줄지 빈칸" 질문의 부분 답이 될 수 있음(중요 정보는 선택 단계가 아니라 이 팝업으로 미룬다).

## 카테고리: 인게임 — 지도 / 노드

### #006 방문 완료(비활성) 노드 시각 처리 + 최초 진입 튜토리얼
- **언제/어디서**: 인게임 지도(MapView), 노드 진입 전후
- **감정**: 진행 상태 가독성 부족 + 신규 플레이어 안내 부재
- **사용자 요구 (날 것)**:
  - 한 번 들어가서 **비활성화된 노드는 색상을 회색/흰색으로 완전히 바꿈** (방문 완료가 한눈에 보이게)
  - **게임 전체에서 "처음 들어가는 종류의 노드"**는 진입하면서 **튜토리얼 팝업**을 띄움
- **메모(확인 필요, 종합 단계)**: "처음 들어가는 노드"의 단위가
  - (A) 노드 *종류*별 최초 1회 (전투 첫 진입 / 이벤트 첫 진입 / 상점 첫 진입 / 엘리트 / 보스 / 활동 등 kind별)인지,
  - (B) 말 그대로 게임 통틀어 가장 처음 밟는 노드 1개인지
  해석 필요. 문맥상 (A) 종류별 최초 1회가 자연스러움(튜토리얼 = 그 종류 설명).
  → 이러면 "튜토리얼 팝업 세트"를 노드 kind마다 1개씩 작성해야 하는 콘텐츠 작업이 따라붙음. 표시 여부 플래그는 메타 진행도(영구 저장)에 기록 필요.
- **연결**: #005 브리핑 팝업과 함께 "신규 플레이어 온보딩" 큰 묶음으로 갈 수 있음.

### #007 시작 노드가 "이미 입장 처리" 되어 입장 불가 (버그성)
- **언제/어디서**: 런 최초 시작, 일루네온 광장(시작 노드)
- **감정**: 어색함/혼란 — 시작하자마자 첫 노드에 못 들어감
- **사용자 요구 (날 것)**:
  - 시작 노드(일루네온 광장)가 **이미 입장(방문) 처리된 상태**라 입장이 막혀 있음 → 이상함
  - 첫 노드가 **마을일 경우, 마을에 들어가 있는 상태로 게임을 시작**하는 게 맞음
- **메모(종합 단계)**:
  - #006(방문=회색/비활성)과 직접 충돌·연동되는 지점. 시작 노드를 "방문 완료"로 칠지 / "현재 머무는 곳"으로 칠지 상태 정의가 필요.
  - 일반화 질문: 마을 노드는 "입장=이벤트 소모형"이 아니라 "체류 가능(상시 재진입)"이어야 하나? 시작 마을만 예외인가, 모든 마을이 그러한가? → 종합 단계에서 마을 노드의 상태 모델 정의.
  - 추정 원인(미검증, 작업 단계에서 코드 확인): 시작 시 현재 노드를 "방문 완료"로 마킹하면서 재진입 차단 로직이 같이 걸리는 구조일 가능성.
- **사용자 확정(2026-06-20)**: 마을 노드가 상시 재진입 가능할 필요는 **없음**. 맨 처음에만 **입장된 상태**로 시작하면 됨.
  - 즉 시작 노드 = "이미 한 번 들어가 있는(소비된) 상태"로 런을 연다. 다만 그게 "바로 입장 막힘 + 회색"처럼 보이면 안 되고, 자연스럽게 "여기서 시작" 느낌이어야 함.
  - #006과의 충돌 해소 방향: 시작 노드는 첫 입장 처리를 *내부적으로 끝낸 채* 시작하되, 첫 화면에서 그 마을 콘텐츠(상점/이벤트 등)를 한 번 보여주고 닫으면 그 다음부턴 회색 비활성으로 가는 게 자연스러울 듯(작업 단계에서 코드 보고 확정).

## 큰 기획: 게임 코어 루프 + 노드 세트 진단 (2026-06-20)

### 노드 4세트 구획 (사용자)
- **전투 세트**: 전투 / 엘리트 / 보스 — *다른 트랙에서 제작 중, 이 논의에선 보류.*
- **거점 세트**: 상점 / 공방 / 마을
- **분기 세트**: 휴식 / 사건
- **수집 세트**: 채집 / 활동

### 코어 루프 (사용자 정의 — 게임의 축)
- 지금 게임 = **보스를 잡는 게임**. 따라서 주된 목표 = 전투 세트 성능을 위한 **캐릭터 강화**.
- **캐릭터 성능 ≈ 카드 성능.** 카드를 강화하는 것이 곧 캐릭터를 강화하는 길.
- 카드 강화 3경로: ① 더 강한 카드를 구한다 ② 카드 자체를 강화한다 ③ 컬러로 강화한다.
  - 아이템 = 임시방편(주된 방향 아님, 단 중요하지 않다는 뜻은 아님).
- **결론(방향성의 핵심)**: "카드 강화를 쉽고 직관적으로" 만드는 것이 게임 방향을 정하는 데 가장 중요.

### 세트별 목표/진단 (사용자)
- **전투** = 실제로 시험하는 곳(proving ground). (보류)
- **거점** = *모든 강화를 담당*. 문제 = 기능이 **애매하게 나뉘어져** 있어, 막상 카드를 강화하려 할 때 **어디로 가야 할지 모름**. (상점/공방/마을 입구 셋의 역할 경계가 불명확)
- **분기(휴식/사건)** = 랜덤 이벤트를 더하지만 **현재 영향이 미미**.
- **수집(채집/활동)** = 중요한 아이템을 주긴 하나 "**전투 보상을 강제로 분리**한 느낌". 
  - 활동은 그 자체로는 괜찮음 — 다만 **노드로 아예 분리되어야 하는지**가 고민.

### 내가 더할 관점 (확인 대기)
- 핵심 긴장 = "강화의 주체(거점)인데 **입구가 3개로 쪼개져** 플레이어의 멘탈 모델('카드 강화하러 어디 가지?')에 1:1로 안 맞는다."
- 강화 3경로 ↔ 노드 매핑이 **여러 노드에 번져(smeared)** 있는 게 혼란의 근원일 가능성. (정확한 현행 터치포인트는 코드로 확인 필요 — 추측 금지.)
- 수집에 대한 사용자 직관("전투 보상 분리")이 날카로움: 채집은 강화 자체가 아니라 **강화의 전(前)자원**(시간조각·골드)을 주는 단계 → 거점으로 흘러듦. "왜 별도 노드인가" 질문 타당.

### 전략 분기: 노드를 어떻게 할 것인가 (사용자, 2026-06-20)
- **경로 A — 통합**: 멘탈 모델과 일치하게 *노드를 줄이고* 한 노드의 역할을 키운다.
- **경로 B — 분화**: 각 노드가 *유의미한 고유 가치*를 지니게 가치를 분리한다.
- 제약/선호: 현 게임은 **월드맵 기반 + 많은 노드 통과를 전제**. → 구조상 **B가 어울림**.
- 사용자 우려: **B가 기획 능력적으로 가능한 일인지** 자신 없음.

### 내 리프레임 (이 분기에 대한 관점)
- **A와 B는 배타 아님.** B가 어려운 이유 = "유한한 가치 파이"를 **너무 많은 슬롯(비전투 9종)** 이 나눠 갖기 때문. 슬롯이 많을수록 각자 몫이 얇아져 필러가 됨(현재 증상).
- → **enabling move = 노드 역할 수를 충분히 줄여(A) 살아남은 노드가 각자 또렷한 가치를 갖게(B) 한다.** A를 B의 수단으로 쓰는 하이브리드. STS가 적은 노드 종류로 B를 깔끔히 해내는 이유가 이것.
- **능력 우려 해소**: 노드 분화는 "고유한 재미 6개를 발명"이 아니라 **가치 축(matrix)에 슬롯 배정**으로 *체계화* 가능 → 천재성이 아니라 프레임의 문제. 이 스캐폴딩은 내가 깔 수 있음(사용자 혼자 짊어질 일 아님).
- 열린 질문(미답): "많은 노드 통과" 전제 자체가 가치인가, 아니면 상속된 가정인가. 필러가 많다면 전제가 *문제의 일부*일 수도. (밀도 높은 작은 맵도 테이블 위 선택지.)

### 동료 시스템 — 세 프레임을 관통하는 누락 변수 (2026-06-20)
- 현황(메모 기반): 로스터 무제한·활성 슬롯 3, 패시브 4종 + 액티브 스킬 ~50, 친밀도 cross-run 영속, 영입 대부분 권역 사건. NPC=몬무스/아인종, 보이스·관계.
- **핵심**: 동료 = 카드와 **별개의 제2 강화 축** + 게임에서 **가장 두꺼운 비전투 콘텐츠 저수지**(사람·친밀도·보이스·영입 서사).
- 세 프레임 관통:
  1. **코어 루프**: 동료는 카드 아닌 또 다른 전투 강화 벡터 → "캐릭터 성능 ≈ 카드 성능" 명제를 *살짝 흐림*. 깨끗한 제2 축이냐 희석이냐가 선택.
  2. **노드 분화**: 영입이 사건/마을에서 일어남 → 약한 세트(사건·마을)에 *지금 당장* 비전투 가치를 줌("사람을 만난다/모은다"). 단 엔딩까지 축적되지 않아 미발현.
  3. **다중 엔딩**: 동료 = **유대/사회 종착 가치의 자연스러운 척추.** gather/activity보다 훨씬 감정적으로 두꺼운 연료.
- **유력 프레이밍(후보, 확정 아님)**: **두 기둥** — 카드=전투 종착, 동료=유대 종착. 2엔딩을 이미 만든 최대 투자 2개(카드/동료)로 떠받치고, 약한 세트는 둘 다에 연료 대는 공용 경제로. → "모든 노드가 기둥이어야" 압박이 풀림.
  - 대안 프레이밍: 동료를 기둥이 아니라 *두 엔딩 모두를 채색하는 가로지르는 레이어*로.
- 정직한 긴장: (a) 전투력 희석(카드 초점), (b) 지금 동료가 전투에서 *실제로 체감되는가*(투자했지만 저(低)체감일 위험 — gather/event와 같은 병), (c) 유대 경로의 보스 답.
- 미확인(코드 검증 필요): 현행 동료 전투 임팩트 실측. 추측 금지.

### 컬러를 척추로 — 노드 재배치 프레임 (사용자, 2026-06-20)
- **유의: 컬러는 경유된 강함(conduit)이지, 문제를 푸는 주체(agent)가 아니다.** 전투가 이미 그러함 — 아무리 강해져도 *전투는 해야* 한다. 컬러만으로 모든 게 풀리면 안 됨.
- 컬러를 경유해 노드를 재배치하기 위한 3요건:
  1. 전투 루트·비전투 루트 **둘 다 컬러를 강화(입력)** 해야 함.
  2. 전투 루트·비전투 루트 **둘 다 컬러를 사용(출력)** 해야 함.
  3. 전투 루트·비전투 루트는 각각의 **"미션"** 을 가짐.
- **유의(패널티)**: 전투의 긴장·보상은 HP + 런 중간 이탈(=실패)이라는 큰 패널티에서 옴. 비전투를 동격 루트로 만들면, **"전투 패배 → 런 실패"를 간접 패널티로 너프**(예: 시간 10턴 경과)해야 함.
- 엔딩: 현재는 "시간의 닻 처치"뿐 — 어쩌면 **싸우지 않는 결말**도 가능. (루트별 미션 = 그 루트의 종착 행위.)

### 내 합성(이 프레임에서 끌어낸 관점)
- 사용자의 "전투 패배 → 시간 패널티"는 사실 **TIME을 공용 스테이크 통화로 세우자**는 제안. → 대칭 구조:
  - 공용 **역량** 기질 = **컬러**(두 루트가 쌓고/쓴다).
  - 공용 **스테이크** 통화 = **시간**(두 루트가 모두 시간을 걸고 잃는다, 런 종료 = 타임아웃).
  - 각 루트는 자기 **미션(행위)** 을 더한다. → 컬러=경유, 미션=주체.
- 정직한 리스크: 시간 스테이크가 *실제로 아파야* 함(300턴 중 10턴은 미약할 수 있음). HP는 "죽음(런 종료)" 대신 "후퇴 → 시간 손실"로 재해석하면 HP의 전술적 역할은 살리고 종결성만 제거 가능.

### 다음: 노드 역할 파편화 정리 착수 → 현행 사실 감사(audit) 완료
전체 보고서 = `node_audit.md`. 핵심 사실(가설 정정 포함):
- **전투 패배 = 즉사 아님**: 목숨 2/2, 패배 시 -1 + 도망(HP30% 회복·노드 미클리어 재도전), 0에서 런 종료. (사용자가 너프하려던 "패배=실패"는 이미 절반 완화됨.)
- **VIT→최대HP 은퇴(F5)**. 현행 컬러→전투 = ATK(불·전기 floor/33 damage)/DEF(흙·철 block)/MAG(빛·어둠 mana)/물→드로우/바람→이동, **전투 시작 1회 정적 변환**. 전투 중 컬러 소비/판정 없음.
- **채집은 컬러 무관**(미니게임 점수만, 임계 상수 0.55, tier는 보상 등급에만). **활동만** 컬러 판정(성공률=clamp(컬러+15)).
- **카드 강화 XP = 전투 전용**(일반1/엘리트3/보스9). 비전투 XP 0. → 워크샵 각성은 별도 통화(특산물·재료).
- 컬러 IN: combat/elite/boss/event/gather/activity. (village·shop·workshop·rest = 컬러 IN 0). 노드 진입 = 시간 -1(전투/비전투 차등 없음), 100턴=하루, 타임아웃 즉시 종료.

**진단(지도)**: 9노드는 기능상 3덩어리 — ①시험(combat/elite/boss) ②컬러·자원 생성기(event/gather/activity) ③변환기(village/shop/workshop+rest). "거점 어디 가지?" = ③ 변환기 3문 중복. 가장 깊은 문제 = **카드 강화 주통화(XP)가 전투 전용이라 비전투는 코어 루프(카드)에 0 기여**. 컬러는 다입력이나 출력 빈약(전투=수동 스탯, 활동만 능동).

### 비전투 = 생활(life-sim) 장르 + 전투 노드 = 인정 게이트 (사용자 결정, 2026-06-20)
- **비전투 = "기대감의 장르"**: 몇 턴 뒤 뭐가 나오고 → 뭘 심고 → 내가 뭘 하고, **계획을 세우고 그게 성립되는 동안 재산이 쌓이는** 느낌. 농사가 원형(가장 직관적). 생활 요소 하나하나가 각자 도움.
  - 채집이 이미 **미니게임 단위로 분화** → 각 미니게임 = 다른 생활 요소로 볼 수 있음. **이걸 늘린다(생활 요소 family 확장).**
- **결정: 비전투는 컬러만 강화**(= 크럭스 (가) 채택). 카드 XP는 전투 고유로 남김.
- **전투 노드 ≠ 비전투가 피하는 것**(파인애플 해소). 전투 노드의 개념 = **"실력 인정" 또는 "우호"**. → 노드 진입 시 **어떤 요소로 인정받을지 선택** 가능해야 함.

### 내 합성 (이 결정들이 만드는 전체 구조)
- **전투 노드 = 다수단 인정 게이트.** 진입 시 수단 선택: 전투(힘) / 실력 인정(컬러·생활 숙련 시연) / 우호(선물·호의 → 친구). 어떤 빌드도 "들어가면 안 되는 노드"가 없음 → **파인애플 완전 해소.**
- **시간 = 진짜 보편 매질.** 전투 = 시간을 *위험*으로 소비 / 생활 = 시간을 *투자*로 소비(심고 기다려 수확). 게이트 = 마감(보스) 압력. → 게임 전체 = "300턴을 build(생활) vs prove(게이트)에 어떻게 배분하나".
- **컬러 출력 문제 해소**: audit에서 컬러 출력이 얇았는데(전투=수동 스탯/활동만), 인정 게이트가 **컬러를 능동 소비**(비전투 수단으로 인정받기)하면서 비전투 루트의 컬러 출구가 생김.
- **우호 수단 = 동료/친밀도 축과 직결**: 게이트에서 적을 친구로 = [[동료 시스템]] 의 비전투 종착. (앞서 논의한 "동료=제2 축"이 여기로 합류.)

### 정직한 도전 2 (여기서 승패가 갈림)
1. **게이트 수단 간 밸런스** — 전투/실력/우호 각각 *실질 비용·트레이드오프*가 있어야. 우호가 늘 쉬우면 전투 사망, 전투가 늘 최선이면 나머지는 장식.
2. **생활의 지속성 vs 맵 churn** — 농사·계획은 *지속되는 터*가 필요(심고 며칠 뒤 수확). 그런데 현행 `advanceDay`는 비보호 노드 content를 매일 재추첨 → 심은 게 하루 지나면 사라질 수 있음. 생활 노드/터의 **영속 모델**을 새로 정의해야 함.

### 인정 게이트 — 수단 메뉴 1차안 (ㄱ, 2026-06-20)
- 진입 화면 = 자리의 존재 + **인정 수단 메뉴**(각 수단의 전제·비용·가능 여부 표시).
- 3 수단 1차안:
  - **전투(힘)**: 항상 가능. 소비 HP·카드. 성공 = 통과 + **XP(카드 성장)** + 드롭 + 컬러. 실패 = 목숨-1·도망(시간 손실). → *유일한 카드 성장로*.
  - **실력 인정(시연)**: 관련 컬러 ≥ 바. 소비 컬러 다발. 성공 = 통과 + 컬러/특산물·평판(**XP 없음**). 실패 = 시간 손실. → 안전·빠름, 카드 안 큼.
  - **우호(호의)**: 특산물/친밀도 보유. 소비 특산물·친밀도. 성공 = 통과 + **동료화 기회**. 실패 = 시간 손실·관계 악화. → 동료 획득로([[동료 시스템]] 합류).
- **핵심 원리**: 수단마다 *비용*만 다른 게 아니라 **"보상 정체성"이 다름** → "무엇이 최선"이 아니라 "지금 내가 뭘 원하나"로 고름 → 밸런스 자동. 루트가 서로 스며듦(전투형도 우호로 HP 아끼고, 생활형도 전투로 카드 키움).

### 게이트 수단 — 단순화 (사용자 정정, 2026-06-20)
- **수단 선택 화면에 flavor 텍스트 금지**(건조하게). "주는 게 다르면 전투 노드 역할이 너무 복잡" → 보상 정체성 차등 폐기. 게이트는 단순.
- 수단 (현재 3개, **컬러 수단 없음**):
  1. **전투** — 기존 전투.
  2. **납품 / 납품 약속** — 몇 턴 뒤까지 특정 아이템을 가져오기로 *약속*. 납품은 같은 노드 아니어도 **마을 같은 곳에서 한 번에** 가능. **납품 실패 시 패널티.**
  3. **그냥 지나치기** — 아무것도 안 하고 통과 가능(비용/보상 없음). → 어떤 게이트도 하드월 아님(파인애플 완전 해소).
- **컬러는 게이트 통과 수단이 아님.** → 컬러의 자리는 게이트 밖으로: 전투 스탯 + (가능성) 생활 효율(농사/채집 성공). 게이트에선 안 씀.
- **의미**: 게이트가 "지금 싸운다 vs 약속하고 나중에 납품한다"로 단순화. 납품 약속 = 외상 통과(약속하고 진행) + 마감 의무. → **생활(농사/채집)에 목적을 줌**(무엇을 길러야 하는지 = 계약). 기대감 장르(계획→누적→납품)를 게이트가 구동. 마을 = 납품 이행 허브(거점 통합과 연결).
- 미결: (a) 납품 실패 패널티 = 무엇(시간? 평판? 게이트 재차단?) (b) 약속=즉시 통과(외상)인가 vs 납품해야 통과인가 (c) 컬러의 자리 — 컬러가 생활(납품재 생산/채집)의 효율을 정하면 "컬러=경유" 유지(색→생활효율→재화→납품→통과). 우호/동료는 납품 부산물(친밀도)로 흡수 가능.

### 농사 시스템 — 상세 설계 (사용자 2026-06-20, "꽤 큰 작업")
- **씨앗(seed)**: 심어야 자람. **무료**(상점 X, 어디서 사는 것 아님). 역할 = *결과물을 구별하는 선택자*(어떤 작물을 기를지).
- **한 씨앗 → 결과물 2종**: 하위(下) / 상위(上).
  - **상위 확률**: 처음엔 낮고 **후반으로 갈수록 상승.** (= 수련회 메타.)
  - **상위는 하위의 상위호환**: 하위를 요구하는 *모든 곳*에 상위를 대신 사용 가능.
- **씨앗마다 요구 턴 수 다름**(grow-time seed-specific).
- **물(water)**: 가끔 중간에 요구. **물을 줘야 나머지 턴이 진행됨**(안 주면 성장 정지). 물 요구 턴은 **랜덤 아님(결정적)** → 계획 가능.
- → 사이클: 씨앗 선택(무료) 심기 → N턴 성장(중간에 결정적 물 체크포인트에서 정지, 물 줘야 재개) → 수확(하위/상위) → 재심기.
- **확인/추천 필요(미결)**:
  1. **상위 확률의 동력**: (a) 전역 런 진행(턴)인지 (b) **농사 스킬 성장**(많이 지을수록 상승)인지. → 추천 (b): 우리가 세운 "활동 비대체적 자기성장"과 정합. "후반=상위↑"는 그때쯤 많이 지어서 자연 상승.
  2. **물의 정체**: 소모 자원인지 vs **단지 그 자리에 와서 주는 무료 행위**인지. → 추천 무료 행위(씨앗도 무료라 일관, 비용은 "제때 돌아오는 동선/시간"). 옵션: 물 색/낚시와 연결(상호 활동 고리).
  3. 결과물 용도 = 납품 물자(추정) + 수확이 컬러(흙·물)도 주는지.
  4. 동시 경작 plot 수(여러 농지 노드 동시 진행 = 물 스케줄 관리 깊이 vs 마이크로 부담).
- **우아한 귀결**: (1) 상위 상위호환 → 후반 고확률 상위가 납품을 *초과 충족* → "후반 영향 커짐" 페이오프. (2) 물-정지 = "돌봄"이 강제돼 농사다운/생활다운 손맛.

## 코드 조사 결과: 노드 재진입/소비 모델 (2026-06-20, run.ts + MapView.vue 확인)

질문 "기존 상시 재진입 가능한 노드가 있었나?" → **있음.** `getEnterAction()`(MapView 187~227) + `advanceDay()`(run.ts 455~) 기준:

**[정정 2026-06-20] "상시 재진입"은 틀린 표현이었음.** `getEnterAction()` 191줄에 `if (node.id === currentNodeId) return 'pass-only'` 가 있어서, **지금 서 있는 노드는 종류 불문 재진입 불가**('지나간다'만 가능). 마을/상점도 용무 보고 나오면 그 자리에 서 있으므로 다시 못 들어감. 한 칸 이동 후 되돌아와야(=다른 턴) 재진입됨.

**A. 소비 상태가 없는 노드(이동 후 돌아오면 재진입 가능, 같은 자리에선 pass-only로 차단)**
- **shop**: done 상태 없음. 카오스 shop-limit(닫힌 시장)만 추가 게이트.
- **village**: 소비 상태 없음.
- **workshop**: 소비 상태 없음.
- → 이 셋은 "방문 후 영구 비활성"이 아니라, *pass-only(같은 자리 차단) + 이동 비용으로* 재진입이 자연 제어됨.

**B. 1회 소비 후 하루 경과 시 리셋(=영구 비활성 아님)**
- combat/elite: `combatCleared` → 'pass'(자동 통과). advanceDay가 풀고 적 재추첨.
- event: `eventTriggered` → 'event-pass'. advanceDay 리셋.
- rest: `restDone` → 'rest-done'(차단). advanceDay 리셋.
- gather: `gatherDone` → 'gather-done'. advanceDay 리셋.
- activity: `activityDone` → 'activity-done'. advanceDay 리셋.
- (advanceDay = 30칸 방문마다(TURNS_PER_DAY). village/boss/shop/workshop은 protectedKinds라 리셋 제외.)

**C. 현재 시각 처리(이미 존재)**
- visited → node-dot opacity 0.7, cleared → 0.35, done → 0.45 (은은한 디밍).
- 카오스 잠금 노드만 *회색*(`--locked`, opacity 0.5). 즉 "확실한 회색"은 지금 카오스 잠금 전용.

### #008 (#006 파생) 재진입 규칙 = 이미 코드와 일치. 회색은 소비형 노드에만.
- **사용자 확정 모델(2026-06-20)**:
  1. **다른 턴에 진입(이동 후 복귀): 재진입 가능**
  2. **같은 자리에서 용무 보고 나옴: 재진입 불가**
  - → 이 둘은 현재 `pass-only`(currentNodeId 차단) 동작과 **이미 일치.** 재진입 관련 별도 코드 변경 불필요(검증만).
- **회색 처리(#006) 범위 정정**:
  - 마을/상점/공방 = **회색 불필요**(사용자 명시). pass-only + 이동비용으로 재진입 제어로 충분.
  - 회색/흰색 비활성 표시는 **소비형 노드의 소비 상태**(combatCleared/eventTriggered/restDone/gatherDone/activityDone)에만 적용. advanceDay 리셋 시 컬러 복귀.
  - 키는 `visited`가 아니라 *소비 상태*. (visited면 마을/상점도 회색돼서 틀림)
- **남은 확인(작업 단계)**: 현재 소비형 노드 회색이 "은은한 디밍(opacity 0.35~0.7)"뿐이라 사용자가 원하는 "확실한 회색/흰색"으로 강화 필요. 카오스 잠금용 `--locked` 회색 스타일을 소비 상태에도 확장하는 방향.

### #007 갱신 — 시작 노드 문제의 진짜 원인 = pass-only
- **원인 확정**: 런 시작 시 `currentNodeId = 시작 노드`(run.ts 275~276 visited 마킹) → `getEnterAction`이 'pass-only' 반환 → "보이는데 못 들어감". 버그가 아니라 pass-only가 시작 노드에도 그대로 적용된 결과.
- **해법(사용자 확정)**: 시작 노드가 마을이면 **마을에 입장된 상태로 런을 연다**(누르게 하지 않음). 즉 런 시작 직후 VillageView를 1회 자동 진입시키거나, 시작 마을 콘텐츠를 한 번 보여준 뒤 맵으로. 그 뒤로는 pass-only 정상 흐름.
- 회색 만들 필요 없음(마을은 회색 대상 아님).
