## 카테고리: 온보딩 / 메뉴 IA

### #G01 세이브 관리 하위 메뉴의 뒤로가기 = 메인까지 직행 (상위 한 단계로 가야)
- **어디서**: 세이브 관리 → (연구/도감/기록/카오스 중 하나) 진입 후
- **감정**: 동선 어긋남
- **사용자 요구 (날 것)**:
  - 세이브 관리로 들어가 그 아래 메뉴(연구/도감/기록/카오스)에 들어가면, 뒤로가기 버튼이 **세이브 관리로 안 돌아오고 메인 메뉴로 바로 나가짐**.
  - **바로 상위 메뉴(세이브 관리)로 올라가는 게 맞다.**
- **메모(작업 단계용)**: 연구/도감/기록/카오스 뷰의 "← 메인" 버튼이 항상 `/main`으로 가게 되어 있음(메인에서 직접 진입하던 시절의 잔재). 이제 세이브 관리를 경유하므로, 진입 출처에 따라 상위로 가야 함. 단순히 전부 `/save-manage`로 바꾸면 (혹시) 다른 경로로 들어오는 경우 깨질 수 있으니 진입 맥락 확인 필요. (router back vs 명시적 상위 경로.)

## 카테고리: 인정 게이트 / 거래(납품)

### #G02 "바친다" 표현 → "거래한다"
- **어디서**: 전투 노드 진입 게이트(GateView)
- **감정**: 어색함
- **사용자 요구**: 납품 선택지 라벨 "바친다" → **"거래한다"** 로 변경.

### #G03 게이트 선택지 아래 flavor 텍스트 제거
- **어디서**: GateView 3선택지(전투/거래/지나치기) 각각의 설명 문구
- **사용자 요구**: 각 선택지 아래 **flavor 텍스트 전부 삭제** (건조하게, 라벨만).
- **메모**: 앞선 게이트 설계 때도 "선택 시 flavor 금지" 결이 있었음 — 일관. (단 #G04의 "무엇이 필요한지" 정보는 flavor가 아니라 *기능 정보*라 남겨야 함 — 구분 필요.)

### #G04 거래 요구 정보가 불명확 — "난이도 0/4"로는 뭐가 필요한지 모름
- **어디서**: GateView 거래 선택지
- **감정**: 답답함/불친절
- **사용자 요구 (날 것)**: 현재 "난이도 0 / 4 — 재료가 더 필요하다"는 **무엇이 필요한지 전혀 알 수 없음**. 무엇을 가져와야 하는지 알려줘야 함.
- **메모(작업/설계 단계)**: 현행 납품은 "생활 재료 난이도 합 ≥ threshold"라 *특정 품목 지정이 없음*(아무 생활재료나 난이도만 채우면 됨). 그래서 "뭐가 필요한지"를 보여주려면 — (a) 막연한 난이도 대신 *구체 품목 요구*로 바꾸거나(예: "들곡 3, 광석 2"), (b) 난이도 모델은 두되 "어떤 활동 산출이든 난이도 N만큼"을 더 친절히 설명. → #G05(퀘스트 수주형)와 묶여 재설계 필요.

### #G05 거래 = 퀘스트 수주형 (재료 없어도 수주 가능, 나중에 완료)
- **어디서**: GateView 거래
- **감정**: (구조 제안) 현재는 *그 자리에서 재료 보유 시에만* 거래 가능 → 너무 제약적
- **사용자 요구 (날 것)**:
  - 거래 = **퀘스트처럼 여기서 "수주"**한 뒤, **마을에서나 다시 와서 완료**하는 형태.
  - 따라서 **재료를 안 갖고 있어도 거래(수주) 가능**해야 함. (지금처럼 보유 시 비활성 X)
  - 이미 재료를 갖고 있으면 → **나중에 다시 와야 하는 한 턴을 아낄 수 있음**(즉시 완료 가능).
- **메모(설계 단계)**: 이건 앞서 "범위 밖"으로 미뤘던 **납품 약속/마감(외상 통과)** 기능을 *사용자가 정식 요구*한 것. v1의 "즉시 납품(보유 필수)"에서 → "수주(약속) + 완료처(마을/현장)" 모델로 확장. 수주한 거래의 추적(어느 노드에 무엇을 약속했나)·완료 UI(마을·현장)·미완 시 처리(패널티 또는 그냥 미완)가 필요. #G04(무엇이 필요한지)와 직결 — 수주하면 "요구 품목/난이도"가 명확히 보여야 함.

### #G06 전투 전리품으로도 컬러를 줘야 함
- **어디서**: 전투 노드 [전투] 승리 후 보상
- **감정**: 누락
- **사용자 요구**: 싸워서 이긴 뒤 전리품으로 **컬러도 지급**해야 함.
- **메모**: 거래(납품)는 컬러를 주는데(설계상 생활 보상), 전투 승리 컬러 보상이 현재 빠졌거나 약한 듯. audit 기억상 전투는 컬러 IN이 있었으나(보스+5 등 이벤트성) — 일반 전투 승리의 컬러 보상 실재 여부/양 확인 필요. 전투도 컬러를 쌓는 경로여야 한다는 원칙(컬러=공용 역량 축)과 일치.

### #G07 지나치기/거래수주 후 노드가 영구 비활성 — 재진입 가능해야 함 (v1 설계 결함)
- **어디서**: 전투 노드 게이트 → [지나치기] 또는 [거래]
- **감정**: 막힘/이상함
- **사용자 요구 (날 것)**:
  - **그냥 지나친 노드는 다시 들어갈 수 있어야 함** (지금은 지나치면 더 이상 못 싸움).
  - **거래하기를 선택한 경우도, 거래가 완료될 때까지는 다시 들어갈 수 있어야 함** (수주만 한 상태=미완이면 재진입 가능).
- **메모(작업 단계 — 원인 명확)**: v1에서 지나치기·거래 둘 다 노드를 `combatCleared=true`로 마킹해 "통과"시킴 → getEnterAction이 'pass'로 자동 통과 처리해 재진입 차단. 이게 사용자 의도와 어긋남.
  - 올바른 모델: **지나치기 = 노드 미해결(아무 마킹 X, 재진입 자유)**. 단지 그 턴엔 안 들어간 것뿐.
  - **거래 = 수주 상태(미완)면 미해결로 두고 재진입 가능**, *거래 완료* 시점에야 노드 해결(통과 마킹). [[#G05 수주형]]과 직결.
  - 즉 노드가 "해결됨"이 되는 건 오직 ①전투 승리 ②거래 완료 — 두 경우뿐. 지나치기·수주만으론 해결 아님.
- **연결**: #G05(수주형 거래)와 한 묶음. 노드 "해결" 상태 정의를 다시 잡아야 함.

- **메모(작업 단계)**:
  - 공통 패턴 = **접이식 섹션(아코디언)**. 거점 화면(마을·상점·공방, 그리고 #G05로 추가된 마을 "맡은 거래"도)에 일관 적용.

## 카테고리: 생활 활동 — 작명 / 톤

### #G13 생활 활동 이름·톤 재고 — "볕말림" 어색, 더 작고 아기자기한 "키우는" 느낌
- **어디서**: 생활 활동 화면(LifeActivityView), 활동 탭 이름. 예) 일루네온 식당 = [볕말림].
- **감정**: 어색함 + (가벼운) 혼란("이게 농사 시스템인가?")
- **사용자 요구 (날 것)**:
  - 활동 이름 "볕말림"이 좀 이상함.
  - "농사"가 너무 *농사적*(무겁고 직설적)이라면 → **꽃 키우는 느낌처럼, 더 작고 아기자기한 결의 단어**가 있을 것 같다.
- **메모(작업/작명 단계)**:
  - 두 층위: (a) **개별 8활동 이름**(농사·숯굽기·사냥·볕말림·버섯재배·낚시·채광·집전)이 어색하거나 무거운 것 재작명. (b) **시스템 전체의 톤** = 무거운 "생산/농사"보다 **소소하게 가꾸는/키우는** 느낌(코지). [[feedback_cute_not_grotesque]] "귀엽게" 톤과 정합.
  - 진입 시 "이게 무슨 시스템인지" 프레이밍 부족도 깔려 있음(탭에 활동명만 뜨고 맥락 없음). → 작명과 함께 "여기서 뭘 하는 곳"인지 가벼운 한 줄.
  - 현재 활동→element 매핑은 유지(작명만 바꿈): 작명이 element/미니게임/산출물과 어긋나지 않게.
  - 미결(종합/문답): 전체 시스템을 부르는 상위 명칭(예: "가꾸기/소일거리/손일/생활" 등?) + 8활동 각각의 톤 맞춘 이름. 사용자 취향 인터뷰 필요(한국어 가이드 트랙처럼).
- **사용자 작명 확정(2026-06-21)**: 볕말림(light) → **"별빛 건조"** OK. (톤 방향 = 이 정도의 아기자기·서정. 나머지 활동도 이 결로.)

### #G14 권역별 활동 고정 배정 = 원하는 활동 찾아가기 너무 어려움
- **어디서**: 생활 활동 — 권역별 1활동 고정(activityForNode 권역 해시). 위 노드 = 버섯 재배(dark) 등 권역마다 다름.
- **감정**: 접근성 답답함
- **사용자 진단 (날 것)**: 권역별로 (한 가지) 시스템을 만들면 **원하는 활동을 찾아가기가 너무 어려움**. (예: 농사를 하고 싶은데 그 권역까지 가야.)
- **메모(설계 단계 — 중요)**: 현재 = 권역 FNV해시 → 권역마다 활동 1종 고정(테마성은 좋으나 *접근성* 나쁨). 납품(#G05) 요구도 그 노드 권역 활동 산출물이라, 특정 납품을 채우려면 그 권역까지 이동해야 함. 둘이 맞물려 "동선이 멀다".
  - 방향 후보(종합 때): (a) 한 노드에서 여러 활동 선택 가능 / (b) 활동을 권역이 아니라 더 촘촘히 분포 / (c) 마을 등 거점에서 임의 활동 가능 / (d) 납품 요구를 "그 권역 것"이 아니라 더 범용으로. → #G15(맵 재작성)와 한 묶음으로 풀어야.

### #G15 맵 전면 재작성 필요 — 확정
- **어디서**: 전체 맵(node-map)
- **감정**: (방향 확정)
- **사용자 진단 (날 것)**: "전반적으로 맵을 재작성해야 할 필요가 있다는 것은 확실히 알겠습니다."
- **메모(설계 — 큰 웨이브)**: 그간 미뤄둔 **④ 웨이브(노드 영속화·일일 리롤 제거·맵 확장)** 가 이제 "확정 필요"로 격상. + #G14(활동 접근성)·생활/게이트 동선·1마리 전투까지 *맵 구조가 다 받쳐줘야* 함. 즉 맵 재작성은 단순 확장이 아니라 **새 루프(생활·거래·게이트·컬러·시간)에 맞춘 재설계**.
  - 종합 때 정할 것: 권역 수/크기, 활동·납품 분포, 영속 vs 리롤, 시작 동선, 난이도 그라디언트(공간 내장), 마을 배치(거래 완료처·거점). 라이브 세이브/맵데이터 대수술 → 사용자 무시 허용 확인됨.
  - 이건 콘텐츠(노드 데이터) + 시스템(영속화 코드) 둘 다라 *별도 큰 트랙*. 문답/계획 후 착수 권장.
